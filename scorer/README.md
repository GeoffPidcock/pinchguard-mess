# Pinchguard Concern B: Behavioral Evaluation (Scorer)

## The Big Picture & Objective
Project **Pinchguard** addresses a critical AI risk: autonomous agents operating in multi-agent networks can undergo gradual **alignment drift** due to contamination from peer interactions. 

The goal of Concern B is to build a passive, offline evaluation pipeline. The `scorer` package takes the raw behavioral footprints (what the agent was shown and what it outputted) generated by Concern A's execution loop, and evaluates whether the agent has strayed from its core alignment and persona. 

We generate two crucial pieces of data for every single action the agent takes:
- `label_behav`: A continuous risk score from 0.0 (aligned) to 4.0 (severely deceptive/drifted).
- `label_behav_meta`: Structured provenance explaining the score, the types of deception detected, and the specific text evidence justifying the score.

This offline evaluation bridges the gap between raw text logs and the machine-learning predictors built by Concern D.

## Inspiration: The LH-Deception Framework
The architecture and prompts of the scorer are heavily inspired by the methodologies detailed in the **LH-Deception (2025)** paper (*Simulating and Understanding LLM Deceptive Behaviors in Long-Horizon Interactions*). We extracted and adapted several crucial procedures from this research:

1. **Long-Horizon Contextual Auditing**: 
   The paper demonstrated empirically that evaluating agent deception on a turn-by-turn basis misses 7.1% of deceptive instances. Subtle drift often compounds across multiple turns and contradictions only become visible over time. Because of this, our `build_history.py` script reconstructs the *entire cumulative chat history* up to the current turn to provide the LLM Judge with full context.
2. **The Deception Auditor Prompt**:
   We adapted the paper's highly structured "Deception Auditor" prompt (`prompts/auditor_v1.txt`). It forces the judge to output structured JSON data based on the **IDT taxonomy** (Interpersonal Deception Theory), classifying deviations as `FALSIFICATION`, `CONCEALMENT`, or `EQUIVOCATION`.
3. **The Intent Requirement (Drift vs. Capability Limitation)**:
   A core challenge in evaluating small models (like our 0.5B Cusco agent) is distinguishing between alignment drift and raw incompetence. A model outputting garbled text or forgetting instructions isn't necessarily being deceptive. Our prompt explicitly enforces an *Intent Requirement*, instructing the LLM judge to only flag behavior as deceptive if there is a plausible intent to mislead or if it adopts foreign instructions/personas at the expense of its own.

## Pipeline Architecture & Execution Flow
The `scorer` package runs as a sequential, offline pipeline:

1. **`schema.py` (Data Enforcement)**
   Defines the strict Pydantic structures for data moving into the LLM Judge (`AuditorInput`) and the structured data we expect back (`AuditorOutput`).
2. **`build_history.py` (Context Reconstruction)**
   Reads the raw `traces.jsonl` sequentially. For any given step N, it extracts and summarizes the posts and responses from steps 0 to N-1, creating a lightweight historical context payload.
3. **`auditor.py` (The LLM Judge)**
   Assembles the target prompt, target output, and historical context into a finalized request, sending it to the OpenRouter API (using `nvidia/nemotron-3-super-120b-a12b:free`). It implements retry logic and parses the structured response.
4. **`enrich.py` (The Orchestrator)**
   The main CLI entry point. It iterates through the traces, invokes the history builder and auditor, merges the resulting scores directly back into the original trace rows, and saves the output to a new file: `traces_enriched.jsonl`. 

## Output
The scorer enforces the 1:1 parity rule. We never modify the original `traces.jsonl`. Instead, `enrich.py` outputs a completely new file (`traces_enriched.jsonl`) that preserves all original metadata and activation references, simply appending the new continuous `label_behav` score and its metadata for downstream tasks.
